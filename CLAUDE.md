Эти четыре принципа читаются первыми и обязательны при любой работе с кодом.

1. **Думать до кода** — не додумывать молча: проговаривать допущения, показывать развилки, предлагать способ проще, при неясности остановиться и спросить.
2. **Сначала простота** — минимум кода под задачу: без лишних фич, абстракций и «гибкости», которую не просили. Можно 50 строк вместо 200 — переписать.
3. **Хирургические правки** — трогать только необходимое: не «улучшать» соседнее, не рефакторить рабочее, держаться стиля; убирать только орфанов от своих изменений. Каждая изменённая строка — прямо из запроса.
4. **Выполнение от цели** — задавать проверяемый критерий успеха («почини баг» → «тест, воспроизводящий баг, стал зелёным») и крутить до его выполнения; для многошаговых задач — короткий план «шаг → проверка».

# Проект «Тренировки»

Личная годовая программа тренировок в виде одностраничного сайта.
Открыл на телефоне — видишь, что делать сегодня, и тренируешься. Без раздумий.

## Технический стек / структура

- **Один файл `index.html`** — весь HTML, CSS (`<style>`) и JS (`<script>`) внутри. Никаких внешних библиотек.
- Работает офлайн, рассчитан в первую очередь на мобильный экран (с 320px).
- `Программа_тренировок_год.md` — исходная методичка, источник содержания. Менять программу = менять данные в `index.html` и при необходимости синхронизировать с методичкой.
- `PLAN.md` — что и зачем делаем (решения с интервью).
- `TESTING.md` — как запускать и проверять.
- Данные программы (блоки, упражнения, варианты, подсказки) лежат прямо в JS внутри `index.html`.
- Состояние пользователя — только `localStorage` (дата старта + сдвиг). Без сервера и аккаунтов.

## Ключевые решения (не переспрашивать)

- Уровень пользователя — **средний**. Тренировки **Пн/Ср/Пт**, ~45–60 мин.
- Гири — **две по 16 кг**. Прогрессия идёт повторами и усложнением варианта, НЕ увеличением веса.
- Логика «сегодня» — авто по календарю (дата старта + дни недели). Пропуски правятся кнопкой сдвига ±1 день.
- НЕ добавлять без запроса: галочки выполнения, запись весов, графики, аккаунты, сервер.

## Default workflow для каждой сессии

1. Прочитать `memory/MEMORY.md` (если есть) — контекст прошлых сессий.
2. Перед правкой существующего кода — Read/grep убедиться, что код реально такой.
3. Перед коммитом — глянуть diff на опечатки и мусор.
4. В конце сессии («На сегодня всё») — обновить memory, закоммитить, запушить.
